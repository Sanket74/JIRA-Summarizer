Ticket fetcher — top 80 selection
The gatekeeper — decides which 80 tickets enter the pipeline each run
NEW 1 day No blocker
▶
What this does
Produces a ranked list of the top 80 tickets every time the pipeline runs. In production it calls the Jira API. In testing it reads the sample dataset and simulates "today's tickets" using a timestamp window. Same interface either way — the rest of the pipeline doesn't know the difference.
PRODUCTION MODE — Live Jira
Call Jira REST API
Filter: status != Done
Filter: updated >= 24h ago
Sort: priority DESC, updated DESC
Return top 80 ticket IDs + fields
TEST MODE — Sample dataset
Read 10k CSV dataset
Simulate "today" = run_date param
Filter by created_date window
Sort: priority DESC
Return top 80 from that window
Ranking logic (both modes)
Priority score: critical=4, high=3, medium=2, low=1
Recency score: updated/created in last 24h gets +1 boost
Sort by combined score DESC, take top 80
Config flag: mode=live|test and run_date=YYYY-MM-DD (test only)
Input
Jira API creds OR\nCSV path + run_date
Output
top80.json\n[{ticket_id, subject,\nbody, priority,\nqueue, tags[]}]
⚠ Decision needed: confirm the ranking logic (priority + recency) with your support lead before building. If they have a different definition of "top 80" — e.g. by comment activity or unresolved age — this logic changes.
2
Batch summariser — 80 primary summaries
Your working prompt wrapped in a loop with validation and retry
YOU HAVE THE PROMPT ✓ 2 days
▶
What this does
Takes the 80 tickets from Step 1. Feeds each one through your existing prompt. Collects outputs. Runs validation on each. Separates passed vs held. Produces a structured JSON. The prompt itself is unchanged — this just industrialises it.
Loop through top80.json, call LLM with your system prompt + ticket body per iteration
Validate each output: word count 10–22, schema fields present (ticket_id, summary, word_count)
Retry logic: if validation fails, retry once with a stricter length instruction appended. If still fails, move to held queue.
Rate limiting + exponential backoff on API errors. Max 3 retries per ticket.
Cost logging: token count per ticket, total batch tokens, estimated cost per run.
Input
top80.json from Step 1
Output
primary_summaries.json\n(passed + held split)\ndaily_cost_log.csv
✓ Acceptance: held rate <5% on a normal run. If held rate >10% two runs in a row — the prompt has drifted against live data distribution. Flag for investigation before continuing.
3
Related ticket summariser — 400 summaries
Same prompt, different input — runs after Sub-problem B delivers related ticket IDs
Needs Sub-problem B output first 1 day
▶
What this does
For each of the 80 primary tickets, Sub-problem B will return 5 related ticket IDs. This step fetches the body of each related ticket and runs it through the same prompt — but with one addition: the summary must also state why it is related to the primary ticket. That link reasoning is the extra requirement for related summaries vs primary summaries.
Take related_tickets.json from Sub-problem B — 80 × 5 = 400 ticket ID pairs
Fetch body of each related ticket from dataset or Jira API
Modified prompt: same 15–20 word rule + append "state why this is related to [primary ticket subject]" instruction
Same validation and retry logic as Step 2
Output pairs each related summary with its primary ticket ID for dashboard linking
Input
related_tickets.json\n(from Sub-problem B)\ndataset or Jira API
Output
related_summaries.json\n{primary_id,\nrelated_id, summary,\nlink_reason}

What this does
